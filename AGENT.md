# qsd Admin AGENT Guide

## 1. 项目定位

这是官网配套的后台管理系统，不是完整物流 ERP，也不是一开始就覆盖订单、财务、结算、审批、风控的全量业务平台。

本后台一期目标是支撑以下能力：

- 官网内容运营
- 新闻发布与维护
- 图片与素材管理
- 客户线索管理
- 运单与轨迹维护
- 客户档案管理
- 账号权限管理
- 操作日志与基础系统设置

后续所有 agent 在构建后台时，必须把“一期范围收敛”作为最高优先级约束，避免后台无限膨胀。

## 2. 一期目标

一期只解决三条核心链路：

1. `官网内容管理`
   - 管理首页、线路页、关于页、联系信息、导航、页脚等官网内容。
2. `客户线索管理`
   - 管理官网表单、咨询线索、分配、跟进与备注。
3. `运单轨迹管理`
   - 管理运单基础信息、轨迹节点、异常标记、签收状态。

一期必须可用，但不追求一次性做成全量经营平台。

## 3. 明确不做

除非用户后续明确要求，否则一期不应主动实现以下能力：

- 订单管理
- 报价管理
- 财务结算
- 发票管理
- 多组织/多租户
- 多仓管理
- 复杂报表中心
- 审批流引擎
- 消息中心
- 风控系统
- 自动化工作流平台

原则：

- 为二期扩展预留边界。
- 不在一期中提前实现复杂中台。
- 不为了“未来可能会用”而增加不必要抽象。

## 4. 后台角色

一期默认只设计三类角色：

- `超级管理员`
  - 拥有所有菜单和权限，负责系统配置、账号、角色、全局数据维护。
- `运营`
  - 负责官网内容、新闻、素材、部分客户线索维护。
- `客服`
  - 负责线索跟进、运单维护、轨迹更新、客户信息补充。

角色设计原则：

- 先做角色驱动的菜单和操作权限。
- 不要一开始设计过细的字段级权限。
- 权限控制优先覆盖“页面访问”和“关键操作”。

## 6. 数据域边界

后台必须按数据域拆分，避免所有逻辑混成一个“大后台模块”。

### 内容域

负责：

- 首页内容
- 线路页内容
- 关于/联系页内容
- 导航、页脚、基础站点信息
- Banner、按钮文案、跳转配置

不负责：

- 客户数据
- 运单状态
- 权限逻辑

### 新闻域

负责：

- 新闻标题
- 摘要
- 封面图
- 正文或外链
- 发布状态
- 发布时间
- 排序

不负责：

- 独立复杂内容工作流
- 评论系统

### 素材域

负责：

- 图片上传
- 文件分类
- 资源 URL
- 资源引用关系

不负责：

- 复杂媒体处理流程

### 线索域

负责：

- 线索来源
- 联系人信息
- 跟进状态
- 负责人
- 跟进记录
- 备注

不负责：

- 正式订单履约
- 财务结算

### 客户域

负责：

- 客户基本资料
- 联系方式
- 历史线索关联
- 历史运单关联

不负责：

- 客户授信
- 合同管理

### 运单域

负责：

- 运单号
- 起始地与目的地
- 当前状态
- 轨迹节点
- 异常状态
- 签收状态

不负责：

- 计费
- 仓储
- 调度

### 权限域

负责：

- 用户
- 角色
- 菜单权限
- 操作权限

### 审计域

负责：

- 登录日志
- 操作日志
- 关键变更记录

## 7. 页面组织原则

后台页面尽量保持统一结构，不要每个模块发明一套不同的交互形式。

推荐页面类型：

- `列表页`
  - 用于查询、筛选、分页、批量操作。
- `详情页`
  - 用于查看实体完整信息。
- `编辑页/抽屉`
  - 用于新增或修改数据。
- `时间线页`
  - 用于展示运单轨迹和跟进记录。
- `日志页`
  - 用于查看系统操作记录。

实现原则：

- 先把 CRUD 做完整。
- 再做高级搜索、批量操作、导出等增强能力。
- 表单优先明确字段校验，不要先追求复杂动态表单引擎。
- 列表和表单组件尽量复用，但不要过度抽象。

## 8. 接口与状态原则

后台不能沿用官网当前“大量静态写死在组件里”的方式。

必须遵守：

- 所有后台实体都要有清晰的数据模型。
- 列表、详情、编辑要围绕真实接口结构设计。
- 表单提交前后要有明确的成功、失败、校验和权限反馈。
- 运单状态、线索状态、新闻发布状态都要有有限状态集合。
- 关键操作必须可记录到操作日志。

不建议：

- 把所有数据塞到一个万能 JSON 字段。
- 直接把官网组件内静态数组搬进后台当正式数据层。
- 在未定义状态枚举前先写页面交互。

## 9. 推荐数据对象

一期至少要明确以下核心对象：

- `admin_user`
- `admin_role`
- `permission`
- `content_page`
- `content_block`
- `news_article`
- `media_asset`
- `lead`
- `lead_follow_record`
- `customer`
- `shipment`
- `shipment_tracking_event`
- `operation_log`
- `login_log`
- `system_setting`

要求：

- 对象命名保持稳定。
- 字段设计优先保证可维护和可审计。
- 能拆表的不要用一张万能表混装所有业务。

## 10. 开发顺序

一期建议严格按下面顺序推进：

1. `后台基础壳子`
   - 登录页
   - 基础布局
   - 路由
   - 权限拦截
   - 全局异常处理
2. `账号权限`
   - 用户、角色、菜单权限
3. `工作台`
   - 基础指标、待办、最近操作
4. `内容管理`
   - 先做首页、联系信息、页脚
5. `新闻管理`
6. `素材库`
7. `线索管理`
8. `运单管理`
9. `客户管理`
10. `操作日志`
11. `系统设置`

开发原则：

- 先打通登录和权限，再做业务页。
- 先做内容和线索，再做运单与客户。
- 先保证可用，再补体验优化。

## 11. 技术与语言强约束

后台项目默认采用以下固定技术栈，除非用户明确要求更换，否则 agent 不应自行改动：

### 管理后台前端

- `Vue 3`
- `TypeScript`
- `Vite`
- `Vue Router`
- `Pinia`
- `Element Plus`

### 后端服务

- `Java 21`
- `Spring Boot 3`
- 标准分层结构：`controller / service / repository(or mapper) / domain`

### 数据与基础设施

- 数据库：`MySQL 8`
- 缓存：`Redis`
- 接口风格：`REST API`
- 文件存储：统一抽象文件服务，允许本地存储或对象存储实现，但对业务层暴露统一接口

### 鉴权

- 基于 `JWT` 或 `Session + Token` 的登录态
- 必须具备角色权限控制
- 必须具备接口级访问控制

语言规则也必须固定：

- 文档说明统一使用 `中文`
- 代码注释统一使用 `中文`
- 提交说明统一使用 `中文`
- 表名、字段名、接口路径、类名、方法名、变量名统一使用 `英文`
- Java 包名使用全小写英文

实现约束：

- 前端不得混用多套后台 UI 框架。
- 后端不得在一期引入不必要的微服务拆分。
- 前端、后端、数据库字段命名必须稳定，避免临时缩写和随意改名。
- 权限、菜单、页面路由结构必须统一。
- 列表页、表单页、详情页优先遵循同一套交互规范。
- 后端接口必须围绕明确的数据对象设计，不能把 mock 结构直接当正式接口。

## 12. 验证要求

每次完成后台改动后，至少验证：

- 项目可以正常构建
- 登录页可用
- 权限拦截有效
- 关键菜单可访问
- 列表页能加载
- 表单校验有效
- 创建/编辑/删除有明确反馈
- 运单轨迹时间线渲染正确
- 操作日志至少覆盖关键写操作

如果涉及角色权限改动，还要验证：

- 不同角色菜单可见性正确
- 无权限用户不能调用关键操作

## 13. 不建议的开发行为

构建后台时，以下行为默认禁止：

- 一期就同时实现订单、财务、审批、报表全量平台
- 没有真实需求就引入复杂低代码抽象
- 用临时 mock 结构充当正式后台数据模型
- 所有页面都写成高度耦合的大组件
- 没有权限模型就先堆业务页面
- 没有日志就直接上线关键维护功能

## 14. 二期扩展方向

如果一期稳定后继续扩展，优先考虑：

1. 订单管理
2. 报价与费用管理
3. 财务结算
4. 报表中心
5. 通知中心
6. 审批流与风控

二期开始前，应重新评估：

- 当前数据模型是否足够支撑扩展
- 是否需要拆分服务边界
- 是否需要更细权限粒度
- 是否需要操作审计加强
